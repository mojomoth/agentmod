Ізоляція та передача проектів для агентів кодування.

`agentmod` зберігає конфігурацію, навички, плагіни, сесії, кеші та
робочий контекст **Claude Code**, **Codex CLI** та **OpenCode** всередині
проекту, над яким ви працюєте — і пакує це середовище в снімок, який
ви можете передати іншій машині.

Він виконує дві ролі:

1. **Agent Home Router.** Всередину дерева каталогів, що містить
   `.agentmod/agentmod.toml`, shell hook маршрутизує кожен дом агента в
   `.agentmod/`. Зовні кожна змінна відновлюється точно так, як вона була, і
   ваша глобальна конфігурація залишається незайманою.
2. **Handoff Tool.** `agentmod handoff create` пакує `.agentmod/` в
   перевірний снімок `.amod` (або з `--for-git`, в дерево файлів, що піддається комітуванню
   під `.agentmod-handoff/`). **Git переміщує ваш код; agentmod переміщує
   середовище агента.**

## Що agentmod це *не*

- **Не пісочниця Docker.** Він маршрутизує змінні середовища у вашій власній
  оболонці. Немає контейнера, немає ВМ, немає фільтрування системних викликів.
- **Не повна ізоляція безпеки.** Інструмент, який ігнорує маршрутизовані змінні,
  може все ще досягти ваших глобальних домів. Claude Bash guard (нижче) — це
  захист в глибину, а не кордон безпеки.
- **Не shimb.** Він ніколи не перехоплює та не обгортає команди `claude`, `codex` або
  `opencode`. Ви продовжуєте запускати їх безпосередньо, без змін.
- **Не інструмент зміни HOME.** `HOME` ніколи не переназивається.
- **Не інструмент для резервного копіювання вихідного коду.** Снімки ніколи не включають
  ваш вихідний код за замовчуванням. Використовуйте git для вихідного коду.

## Як це працює

`agentmod hook zsh` / `agentmod hook bash` виводять невелику самостійну функцію shell
(встановлену у ваш файл rc за допомогою `agentmod init`). На кожному запиті та
змінюванні каталогу він проходить вгору у пошуках `.agentmod/agentmod.toml`:

- **Вхід до проекту** зберігає поточні значення та встановлює:

  | Змінна | Маршрутизовано до |
  |---|---|
  | `CLAUDE_CONFIG_DIR` | `.agentmod/claude` |
  | `CODEX_HOME` | `.agentmod/codex` |
  | `OPENCODE_CONFIG` | `.agentmod/opencode/opencode.json` |
  | `NPM_CONFIG_PREFIX` | `.agentmod/node` |
  | `NPM_CONFIG_CACHE` | `.agentmod/node/npm-cache` |
  | `PNPM_HOME` | `.agentmod/node/pnpm` |
  | `BUN_INSTALL` | `.agentmod/node/bun` |
  | `XDG_CONFIG_HOME` / `XDG_DATA_HOME` / `XDG_CACHE_HOME` / `XDG_STATE_HOME` | `.agentmod/opencode/xdg/…` — **тільки** з `opencode.xdg_full_isolation = true` |

  `PATH` отримує рівно один запис, `.agentmod/node/bin` (глобальний bin npm
  під маршрутизованим префіксом). Змінні ведення записів (`AGENTMOD_ACTIVE`,
  `AGENTMOD_PROJECT_ROOT`, `AGENTMOD_ROOT`, `AGENTMOD_VARS`,
  `AGENTMOD_SAVED_*`) записують, що скасувати.

- **Вихід з проекту** відновлює кожне збережене значення та вилучає
  запис `PATH` — ідеальне обернення. Перемикання безпосередньо між двома проектами agentmod
  перемаршрутизує за один крок без витоку шляхів жодного проекту.

Маршрутизація per agent може бути вимкнена в `agentmod.toml`
(`claude.enabled`, `codex.enabled`, `opencode.enabled`, `node.enabled`).

## Встановлення

Виберіть те, що підходить вашій конфігурації — кожне встановлює той же самий бінарний файл:

```sh
# npm (встановлює попередньо зібраний бінарний файл для вашої платформи)
npm install -g agentmod

# скрипт встановлення (завантажує відповідний реліз, перевіряє sha256)
curl -fsSL https://raw.githubusercontent.com/mojomoth/agentmod/main/install.sh | sh

# go install (вимагає набір інструментів Go)
go install github.com/mojomoth/agentmod@latest
```

Або побудуйте з вихідного коду (Go 1.26+, єдина залежність модуля —
`BurntSushi/toml`):

```sh
git clone https://github.com/mojomoth/agentmod && cd agentmod
go build -o agentmod .
# помістіть бінарний файл десь на вашому PATH
```

## Швидкий старт

```sh
cd ~/work/myproject
agentmod init          # створює .agentmod/, редагує .gitignore, встановлює
                       # hook оболонки у ваш файл rc, пропонує скопіювати auth
# тільки в першому разі: hook ще не активний в цій оболонці —
# відкрийте новий терміналь, або: exec $SHELL

cd ~/work/myproject    # hook активується; перевірте його:
agentmod status        # "AgentMod: active", маршрутизовані шляхи відображаються
claude                 # звичайна команда — тепер використовується локальний дом проекту
agentmod install gstack   # навички локального проекту, глобальний дом без змін

agentmod pack          # снімок до .agentmod/snapshots/<name>-<stamp>.amod
agentmod doctor        # діагностика лише для читання в будь-який час
```

На одержувальній машині:

```sh
cd ~/work/myproject    # вихідний код надійшов через git
agentmod init
agentmod unpack myproject-20260611-123045.amod
# дотримуйтесь надрукованих нотаток повторного входу; doctor запускається автоматично
```

## `agentmod init`

Ідемпотентний — повторне запуск заповнює все, що не вистачає, і ніколи не перезаписує
існуючий `agentmod.toml` або будь-який користувальницький файл. Це:

- створює `.agentmod/{claude,codex,opencode,node,snapshots,logs}` та
  стандартний `agentmod.toml`;
- підключає Claude Bash guard до `.agentmod/claude/settings.json`;
- додає `.agentmod/` до `.gitignore` (створюється лише всередині git-сховища);
- встановлює shell hook як огорджену блок у `~/.zshrc` або `~/.bashrc`
  (ваша оболонка з `$SHELL`; блок оновлюється на місці, ніколи не
  дублюється, і ваш власний вміст rc ніколи не торкається);
- пропонує **скопіювати** існуючі файли автентифікації Claude/Codex у локальний дом проекту
  (див. "Auth" нижче) — копіювання відбувається лише на явне `y`.

Прапори: `--no-shell-hook` пропускає всі редагування файлу rc; `--yes` /
`--non-interactive` ніколи не запитує й, отже, ніколи не копіює auth (для CI).

## Використання звичайних `claude`, `codex`, `opencode`

Немає команди-обгортки. Всередину активного проекту звичайні команди
просто бачать маршрутизовані дома:

- **Claude Code** читає `CLAUDE_CONFIG_DIR` → параметри локального проекту,
  навички/плагіни на рівні користувача, сесії, історія. (Проект `.claude/`
  *завжди* читається нативно — див. Обмеження.)
- **Codex CLI** читає `CODEX_HOME` → локальний `config.toml`,
  `auth.json`, сесії, історія, журнали.
- **OpenCode** читає `OPENCODE_CONFIG` → локальний файл конфігурації.
  Це *часткова* ізоляція за замовчуванням — див. Обмеження.

### Auth

Нові локальні дома проекту стартують без облікових даних:

- **Claude на macOS**: нічого робити — облікові дані живуть в Keychain і
  спільні для кожного каталогу конфігурації (що також означає, що вони *не*
  ізольовані per project).
- **Claude на Linux/Windows**: запустіть `claude login` всередину проекту, або
  прийміть пропозицію init скопіювати `~/.claude/.credentials.json`.
- **Codex**: запустіть `codex login` всередину проекту, або прийміть
  пропозицію init скопіювати `~/.codex/auth.json`.

Файли Auth **ніколи не подорожують в снімки** (виключені за назвою, незалежно від
як вони туди потрапили).

## Встановлення gstack

[gstack](https://github.com/garrytan/gstack) жорстко кодує свій інсталятор на
`~/.claude/skills/gstack` — саме глобальне забруднення, яке agentmod існує для
запобігання. Таким чином:

```sh
agentmod install gstack            # клонування в .agentmod/claude/skills/gstack
agentmod install gstack --force    # заміна існуючого локального встановлення проекту
```

Інсталятор клонує з git, ніколи не запускає власний скрипт налаштування gstack, і
робить снімок переліку `~/.claude/skills` до та після — будь-яка зміна
глобального каталогу повідомляється як порушення та завершує команду.
`agentmod doctor` окремо попереджає, коли існує *глобальне* встановлення gstack
(навіть те, що ви встановили самі до прийняття agentmod), оскільки глобально
встановлені навички витікають у кожний проект.

## Handoff (снімки `.amod`)

```sh
agentmod handoff create [--output PATH] [--allow-findings] [--allow-dirty]
agentmod handoff list
agentmod handoff inspect FILE      # маніфест + звіт редагування, без видобування
agentmod handoff verify  FILE      # повторно гешір кожного члена; вихід 3 при невідповідності
agentmod handoff restore FILE      # заміна .agentmod/ (резервна копія зроблена спочатку)
agentmod pack / agentmod unpack    # псевдоніми створення / відновлення
```

Снімок — це zip з шістьма кореневими членами — `manifest.json`,
`inventory.json` (розмір per-file/sha256/mode), `REDACTION.md` (що
було виключено та чому, плюс знахідки сканування таємниці), `HANDOFF.md` та `RESTORE.md`
(людські інструкції для одержувача), `checksums.txt`
(сумісний з `shasum -a 256 -c`) — та корисне навантаження під
`payload/.agentmod/…`. Створення є атомним та детермінованим; маніфест
записує гілку git/коміт/брудний стан з облікові дані, видалені з URL-адреси
віддаленого. Брудне дерево робочої області відмовляється пакувати, якщо не `--allow-dirty`.

`inspect` та `verify` працюють скрізь — одержувач може провести аудит снімка
перед будь-яким налаштуванням проекту.

### Політика виключення таємниць

Два шари, обидва увімкнені за замовчуванням:

1. **Правила виключення** видаляють известіні чутливі файли з корисного навантаження та
   перелічують кожне в `REDACTION.md`: файли auth за назвою (`.credentials.json`,
   `auth.json`, `credentials*`), `*.env` / `.env.*`, SSH ключі (`id_*`,
   `*.pem`, `*.pub`), каталоги облікових даних (`.ssh`, `.aws`, `.azure`,
   `.gcloud`, `.kube`, `.gnupg`, `.docker`), файли keychain, `.git`,
   `node_modules`, кеші та тимчасові каталоги.
2. **Сканування вмісту** над кожним *збереженим* файлом. Матеріал приватного ключа відмовляється
   створенням без виключення, якщо ви не передасте `--allow-findings` (і тоді позначається
   HARD в `REDACTION.md`). Можливі токени (AWS access key IDs, GitHub
   токени, `sk-…` ключі, призначення стилю `api_key=`) попереджаються, але
   не блокуються.

Сканування — евристичне. **Перегляньте `REDACTION.md` (або `handoff inspect`) перед
спільним використанням снімка** — сесії та робочий контекст подорожують за задумом
і можуть цитувати все, що ви вставили в розмову агента. Снімки записуються
режимом 0600 з цієї причини; обробляйте їх як приватні файли.

## Git handoff

```sh
agentmod pack --for-git    # пише .agentmod-handoff/ у корені проекту
git add .agentmod-handoff && git commit
```

Однакові шість членів та корисне навантаження як `.amod`, але як дерево файлів, що піддається комітуванню
(`shasum -a 256 -c checksums.txt` працює в каталозі). На вершині
стандартних виключень він видаляє **сесії, записи, історія,
та журнали** для всіх трьох агентів — вони звичайно містять вставлені таємниці та
не належать до сховища. `--include-sessions` завжди відмовляється:
комітування сесій вимагало б шифрування, яке цей версія не
реалізує. Робочий контекст, який можна безпечно спільно використовувати (CLAUDE.md, конфігурації агента,
навички, плани) залишається.

Повторне запуск замінює попередній пакет; нічого іншого в сховище не
торкається.

## Попередження при відновленні

`handoff restore` / `unpack` розглядає кожен снімок як недовірені вхідні дані:

- повна перевірка контрольної суми та перевірка інвентарю спочатку;
- план безпеки шляху: zip-slip (`..`), абсолютні шляхи, букви дисків,
  цілі, що не мають `.agentmod`, захищені назви (`.git`, `.ssh`, `.aws`,
  `.docker`), та екранування або абсолютні цілі символічного посилання — все
  відмовляється перед записом чого-небудь;
- існуючий `.agentmod/` перейменовується на `.agentmod.backup-<stamp>` перед
  видобуванням; будь-яка відмова автоматично повертається до нього;
- **нічого з снімка ніколи не виконується**;
- потім: Claude guard hook перепідключена до *цього* бінарного файлу машини,
  машино-специфічні абсолютні шляхи, знайдені у відновлених конфігураціях агента, попереджаються
  (ваші файли ніколи не переписуються), `doctor` запускається вбудовано, та
  друкуються необхідні кроки повторного входу (auth ніколи не подорожує).

Відновлення відмовляються, а не вгадують — відмовлене відновлення залишає проект
байт-ідентичним.

## `agentmod doctor`

Діагностика лише для читання, безпечно запускається в будь-який час (вихід 0 чистий, 3 з знахідками):
стан проекту/конфігурації/макета, встановлення та живість shell-hook, дрейф маршрутизації,
залишкові змінні поза проектами, дублювання записів PATH,
порушення HOME/shim, присутність per-agent auth з інструкціями повторного входу,
попередження витоків OpenCode, стан глобального/проекту gstack, Claude guard wiring,
ризики портативності в відновлених конфігураціях, кандидати таємниць записані в
існуючих снімках, матеріал сесії/журналу всередину `.agentmod-handoff/`, та
чи голова сховища все ще відповідає найновішому снімку.

## Claude Bash guard

`agentmod init` реєструє `agentmod guard claude-bash` як Claude Code
PreToolUse hook у локальному домі проекту. Він блокує команди Bash, які
писатимуть до глобальних домів агента (`~/.claude`, `~/.codex`,
`~/.config/opencode`, `~/.local/share/opencode`), використовувати `sudo`, або
переназначати `HOME` — агент отримує причину назад та може налаштуватися. Читання ніколи
не блокуються. Це один shell-parse евристичний глибокий: корисна огорожа, не
пісочниця.

## Відомі обмеження

Секція честі. Це властивості базових інструментів або навмисна
область MVP — `doctor` та згенеровані документи також їх посилають.

- **macOS Keychain (Claude).** Claude Code на macOS зберігає облікові дані OAuth
  в Keychain, спільні для *всіх* каталогів конфігурації. Ізоляція облікового запису per-project
  неможливо на macOS — та повторний вхід не потрібен per project.
  Linux/Windows використовує per-home `.credentials.json`, що ізолює, але
  вимагає входу/копіювання per project.
- **OpenCode частково ізольований за замовчуванням.** OpenCode немає однієї
  змінної дому; його конфігурація — це ланцюг злиття, який все ще читає глобальну
  `~/.config/opencode/opencode.json`, та сесії/сховище/auth живуть у
  глобальних каталогах XDG. `opencode.xdg_full_isolation = true` маршрутизує
  змінні XDG для повної ізоляції — але це впливає на *кожен* інструмент,
  чутливий до XDG, який ви запускаєте всередину проекту. `doctor` повідомляє обидва
  ситуації.
- **Project `.claude/` — нативна поведінка Claude.** Claude Code завжди
  читає `./.claude/` незалежно від `CLAUDE_CONFIG_DIR`. додана вартість agentmod
  для Claude — ізоляція стану *на рівні користувача* (навички/плагіни глобального рівня,
  сесії, історія); project `.claude/` вже працював перед agentmod.
- **Активація hook першої сесії.** Відразу після `agentmod init`, вже
  запущена оболонка не завантажила новий блок rc. Відкрийте новий
  терміналь, `exec $SHELL`, або one-shot `eval "$(agentmod hook zsh)"` (init
  виводить саме це). Аналогічно, bash hook спалює через `PROMPT_COMMAND`
  та, отже, неактивний у non-interactive bash скриптах (той же клас
  обмеження як direnv) — скрипти повинні встановлювати змінні явно через
  `eval "$(agentmod env --shell bash --activate <root>)"` якщо їм потрібна
  маршрутизація.
- **Тільки глобальний bin npm на PATH.** `.agentmod/node/bin` — це єдиний
  керований запис PATH. pnpm/bun глобальні встановлення маршрутизуються в проект
  (`PNPM_HOME`, `BUN_INSTALL`), але їх каталоги bin не додаються до PATH.
- **Пакети Tree відновлюються вручну.** `handoff restore` приймає лише файли `.amod`;
  комітованого каталогу `.agentmod-handoff/` відновлюється шляхом
  дотримання `RESTORE.md` всередину нього (ця версія немає читача каталогу).
- **Снімки можуть потребувати post-restore ремонту.** Клонування gstack подорожує
  без його `.git` (повторно запустіть `agentmod install gstack --force`, щоб зробити його
  оновлюваним знову), та `node/bin` launcher символічні посилання висять тому що
  `node_modules` виключено (повторно запустіть `npm install -g …` всередину
  проекту).
- **Підтримка Shell — zsh та bash.** Інші оболонки все ще можуть використовувати
  `agentmod env` вручну.

## Частих питань

**Чи я продовжую використовувати `claude` / `codex` / `opencode` безпосередньо?**
Так. Це суть — немає обгорток, немає shims, немає `agentmod run`.

**Чому agentmod просто не змінює `HOME`?**
Переназначення `HOME` порушує SSH, git, keychains, dotfiles, та кожен інший
інструмент в оболонці. agentmod маршрутизує лише змінні, специфічні для агента.

**Чому мій auth відсутній після відновлення?**
За задумом — облікові дані ніколи не подорожують в снімки. Дотримуйтесь надрукованих
ліній повторного входу (або пропозиція init копіювати) на новій машині.

**Чи можу я коммітити `.agentmod/` на git?**
Ні — init gitignores його (сесії, кеші, та можливо копійовані auth живуть
там). Замість цього комітуйте безпечне підмножество: `agentmod pack --for-git`.

**Чим це відрізняється від direnv?**
Та сама модель активації (область каталогу env, prompt-hook based, ідеальне
відновлення при виході), але agentmod також знає *що* маршрутизувати для кожного агента,
створює дома, захищає від глобальних записів, та здійснює handoff. Обидва
співіснують добре.

**Снімок завершується помилкою з "secret-candidate findings".**
Сканування вмісту знайшло матеріал приватного ключа в збереженому файлі. Видаліть його (або
перемістіть його у виключене місцезнаходження на кшталт `.env`), або пакуйте у будь-якому разі з
`--allow-findings`, якщо ви приймаєте його приймання всередину снімка.

**Чи працює це на Windows?**
Go код будується та безпека шляху застосовується для шляхів Windows, але
shell hooks цільові zsh/bash; Windows не тестується в цій версії.
