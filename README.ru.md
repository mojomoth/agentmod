Изоляция и передача проектов для кодирующих агентов.

`agentmod` хранит конфигурацию, навыки, плагины, сеансы, кэши и
рабочий контекст **Claude Code**, **Codex CLI** и **OpenCode** внутри
проекта, над которым вы работаете — и упаковывает эту среду в снимок,
который вы можете передать другой машине.

Она выполняет две роли:

1. **Маршрутизатор Agent Home.** Внутри дерева каталогов, содержащего
   `.agentmod/agentmod.toml`, хук оболочки маршрутизирует домашний каталог
   каждого агента в `.agentmod/`. Снаружи каждая переменная восстанавливается
   точно так, как она была, и ваша глобальная настройка не затрагивается.
2. **Инструмент передачи.** `agentmod handoff create` упаковывает `.agentmod/`
   в проверяемый снимок `.amod` (или, с `--for-git`, в коммитируемое дерево
   файлов под `.agentmod-handoff/`). **Git перемещает ваш исходный код;
   agentmod перемещает среду агента.**

## Что agentmod *не является*

- **Не Docker-песочница.** Она маршрутизирует переменные окружения в вашей
  собственной оболочке. Нет контейнера, нет виртуальной машины, нет фильтрации
  системных вызовов.
- **Не полная безопасность изоляции.** Инструмент, который игнорирует маршрутизируемые
  переменные, все еще может получить доступ к вашим глобальным домашним каталогам.
  Охрана Claude Bash (см. ниже) — это защита в глубину, а не граница безопасности.
- **Не shim.** Она никогда не перехватывает и не оборачивает команды `claude`, `codex`
  или `opencode`. Вы продолжаете запускать их напрямую, без изменений.
- **Не инструмент для изменения HOME.** `HOME` никогда не переназначается.
- **Не инструмент резервного копирования исходного кода.** Снимки никогда не включают
  ваш исходный код по умолчанию. Используйте git для исходного кода.

## Как это работает

`agentmod hook zsh` / `agentmod hook bash` печатают небольшую самодостаточную функцию
оболочки (установленную в ваш файл rc с помощью `agentmod init`). При каждом приглашении
и изменении каталога она поднимается в поисках `.agentmod/agentmod.toml`:

- **Вход в проект** сохраняет текущие значения и устанавливает:

  | Переменная | Маршрутизируется в |
  |---|---|
  | `CLAUDE_CONFIG_DIR` | `.agentmod/claude` |
  | `CODEX_HOME` | `.agentmod/codex` |
  | `OPENCODE_CONFIG` | `.agentmod/opencode/opencode.json` |
  | `NPM_CONFIG_PREFIX` | `.agentmod/node` |
  | `NPM_CONFIG_CACHE` | `.agentmod/node/npm-cache` |
  | `PNPM_HOME` | `.agentmod/node/pnpm` |
  | `BUN_INSTALL` | `.agentmod/node/bun` |
  | `XDG_CONFIG_HOME` / `XDG_DATA_HOME` / `XDG_CACHE_HOME` / `XDG_STATE_HOME` | `.agentmod/opencode/xdg/…` — **только** с `opencode.xdg_full_isolation = true` |

  `PATH` получает ровно одну запись, `.agentmod/node/bin` (глобальный bin npm
  под маршрутизируемым префиксом). Переменные ведения записей (`AGENTMOD_ACTIVE`,
  `AGENTMOD_PROJECT_ROOT`, `AGENTMOD_ROOT`, `AGENTMOD_VARS`,
  `AGENTMOD_SAVED_*`) записывают, что отменить.

- **Выход из проекта** восстанавливает каждое сохраненное значение и удаляет запись
  `PATH` — идеальное обратное действие. Прямое переключение между двумя проектами
  agentmod переже маршрутизирует в одном шаге без утечки путей любого проекта.

Маршрутизация для каждого агента может быть отключена в `agentmod.toml`
(`claude.enabled`, `codex.enabled`, `opencode.enabled`, `node.enabled`).

## Установка

Выберите то, что соответствует вашей настройке — каждое устанавливает один и
тот же единственный бинарный файл:

```sh
# npm (устанавливает предварительно собранный бинарный файл для вашей платформы)
npm install -g agentmod

# сценарий установки (загружает соответствующий выпуск, проверяет sha256)
curl -fsSL https://raw.githubusercontent.com/mojomoth/agentmod/main/install.sh | sh

# go install (требует цепочки инструментов Go)
go install github.com/mojomoth/agentmod@latest
```

Или собрать из исходного кода (Go 1.26+, единственная зависимость модуля —
`BurntSushi/toml`):

```sh
git clone https://github.com/mojomoth/agentmod && cd agentmod
go build -o agentmod .
# поместите бинарный файл куда-то на ваш PATH
```

## Быстрый старт

```sh
cd ~/work/myproject
agentmod init          # создает .agentmod/, редактирует .gitignore, устанавливает
                       # хук оболочки в ваш файл rc, предлагает скопировать аутентификацию
# только в первый раз: хук еще не живой в ЭТОЙ оболочке —
# откройте новый терминал или: exec $SHELL

cd ~/work/myproject    # хук активируется; проверьте его:
agentmod status        # "AgentMod: active", перечислены маршрутизированные пути
claude                 # простая команда — теперь использует локальный домашний каталог проекта
agentmod install gstack   # локальные навыки проекта, глобальный домашний каталог не затронут

agentmod pack          # снимок в .agentmod/snapshots/<name>-<stamp>.amod
agentmod doctor        # диагностика только для чтения в любое время
```

На принимающей машине:

```sh
cd ~/work/myproject    # исходный код прибыл через git
agentmod init
agentmod unpack myproject-20260611-123045.amod
# следуйте напечатанным инструкциям по повторному входу; doctor запускается автоматически
```

## `agentmod init`

Идемпотентный — повторный запуск заполняет все, что отсутствует, и никогда не
перезаписывает существующий `agentmod.toml` или какой-либо файл пользователя. Это:

- создает `.agentmod/{claude,codex,opencode,node,snapshots,logs}` и стандартный
  `agentmod.toml`;
- подключает охрану Claude Bash в `.agentmod/claude/settings.json`;
- добавляет `.agentmod/` в `.gitignore` (создается только внутри репозитория git);
- устанавливает хук оболочки как ограниченный блок в `~/.zshrc` или `~/.bashrc`
  (ваша оболочка из `$SHELL`; блок обновляется на месте, никогда не дублируется,
  и ваше собственное содержимое rc никогда не трогается);
- предлагает **скопировать** существующие файлы аутентификации Claude/Codex в
  локальный домашний каталог проекта (см. «Аутентификация» ниже) — копирование
  происходит только при явном согласии `y`.

Флаги: `--no-shell-hook` пропускает все редактирования файла rc; `--yes` /
`--non-interactive` никогда не выпрашивает и поэтому никогда не копирует
аутентификацию (для CI).

## Использование простых `claude`, `codex`, `opencode`

Нет команды-оболочки. Внутри активного проекта обычные команды просто видят
маршрутизированные домашние каталоги:

- **Claude Code** читает `CLAUDE_CONFIG_DIR` → локальные параметры проекта,
  навыки/плагины уровня пользователя, сеансы, история. (`.claude/` уровня проекта
  *всегда* читается нативно — см. Ограничения.)
- **Codex CLI** читает `CODEX_HOME` → локальный `config.toml`, `auth.json`,
  сеансы, историю, журналы.
- **OpenCode** читает `OPENCODE_CONFIG` → локальный файл конфигурации.
  Это *частичная* изоляция по умолчанию — см. Ограничения.

### Аутентификация

Новые локальные домашние каталоги проекта начинаются без учетных данных:

- **Claude на macOS**: ничего не нужно — учетные данные находятся в Keychain
  и совместно используются с каждым каталогом конфигурации (что также означает,
  что они *не* изолированы для каждого проекта).
- **Claude на Linux/Windows**: запустите `claude login` внутри проекта или
  примите предложение init скопировать `~/.claude/.credentials.json`.
- **Codex**: запустите `codex login` внутри проекта или примите предложение init
  скопировать `~/.codex/auth.json`.

Файлы аутентификации **никогда не путешествуют в снимках** (исключены по названию,
независимо от того, как они туда попали).

## Установка gstack

[gstack](https://github.com/garrytan/gstack) жестко кодирует свой установщик
в `~/.claude/skills/gstack` — именно глобальное загрязнение, которое agentmod
существует для предотвращения. Поэтому:

```sh
agentmod install gstack            # клон в .agentmod/claude/skills/gstack
agentmod install gstack --force    # заменить существующую установку на уровне проекта
```

Установщик клонирует с git, никогда не запускает собственный сценарий setup
gstack, и делает снимок списка `~/.claude/skills` до и после — любое изменение
глобального каталога сообщается как нарушение и не выполняет команду.
`agentmod doctor` отдельно предупреждает всякий раз, когда существует *глобальная*
установка gstack (даже та, которую вы установили сами до принятия agentmod),
потому что глобально установленные навыки просачиваются в каждый проект.

## Передача (снимки `.amod`)

```sh
agentmod handoff create [--output PATH] [--allow-findings] [--allow-dirty]
agentmod handoff list
agentmod handoff inspect FILE      # манифест + отчет редакции, без извлечения
agentmod handoff verify  FILE      # пересчитать каждого члена; выход 3 при несовпадении
agentmod handoff restore FILE      # заменить .agentmod/ (сделана резервная копия)
agentmod pack / agentmod unpack    # псевдонимы create / restore
```

Снимок — это zip с шестью корневыми членами — `manifest.json`,
`inventory.json` (размер/sha256/режим для каждого файла), `REDACTION.md`
(что было исключено и почему, плюс результаты сканирования секретов),
`HANDOFF.md` и `RESTORE.md` (инструкции для получателя), `checksums.txt`
(совместимый с `shasum -a 256 -c`) — и полезная нагрузка под
`payload/.agentmod/…`. Создание атомное и детерминированное; манифест
записывает ветку/коммит/грязное состояние git с любыми учетными данными,
удаленными из URL удаленного. Грязное дерево работы отказывается упаковываться,
если не `--allow-dirty`.

`inspect` и `verify` работают везде — получатель может проверить снимок,
прежде чем иметь какую-либо настройку проекта.

### Политика исключения секретов

Два слоя, оба включены по умолчанию:

1. **Правила исключения** удаляют известные конфиденциальные файлы из полезной
   нагрузки и перечисляют каждый в `REDACTION.md`: файлы аутентификации по названию
   (`.credentials.json`, `auth.json`, `credentials*`), `*.env` / `.env.*`,
   SSH-ключи (`id_*`, `*.pem`, `*.pub`), каталоги учетных данных (`.ssh`, `.aws`,
   `.azure`, `.gcloud`, `.kube`, `.gnupg`, `.docker`), файлы цепочки ключей,
   `.git`, `node_modules`, кэши и временные каталоги.
2. **Сканирование содержимого** для каждого *сохраненного* файла. Материал
   приватного ключа отказывает в создании иначе, если вы не передадите
   `--allow-findings` (и затем помечаются HARD в `REDACTION.md`). Вероятные токены
   (идентификаторы ключей доступа AWS, токены GitHub, ключи `sk-…`, назначения
   стиля `api_key=`) предупреждаются, но не блокируют.

Сканирование эвристическое. **Просмотрите `REDACTION.md` (или `handoff inspect`)
перед обменом снимком** — сеансы и рабочий контекст путешествуют по замыслу и
могут цитировать все, что вы вставили в разговор агента. Снимки написаны режимом
0600 по этой причине; относитесь к ним как к приватным файлам.

## Передача по Git

```sh
agentmod pack --for-git    # написать .agentmod-handoff/ в корень проекта
git add .agentmod-handoff && git commit
```

Те же шесть членов и полезная нагрузка, что и в `.amod`, но как коммитируемое
дерево обычных файлов (`shasum -a 256 -c checksums.txt` работает в каталоге).
Поверх стандартных исключений он удаляет **сеансы, расшифровки, историю и журналы**
для всех трех агентов — они регулярно содержат вставленные секреты и не принадлежат
репозиторию. `--include-sessions` всегда отказывает: коммитинг сеансов требовал бы
шифрования, которое эта версия не реализует. Рабочий контекст, безопасный для
совместного использования (CLAUDE.md, конфигурации агентов, навыки, планы),
остается.

Повторный запуск заменяет предыдущий пакет; ничего другого в репозитории не
трогается.

## Предосторожности при восстановлении

`handoff restore` / `unpack` рассматривает каждый снимок как недоверенный ввод:

- полная проверка контрольной суммы и перекрестная проверка инвентаря в первую очередь;
- план безопасности пути: zip-slip (`..`), абсолютные пути, буквы дисков,
  не-`.agentmod` цели, защищенные имена (`.git`, `.ssh`, `.aws`, `.docker`),
  и экранирование или абсолютные цели символических ссылок — все отказаны
  перед написанием чего-либо;
- существующий `.agentmod/` переименовывается в `.agentmod.backup-<stamp>` перед
  извлечением; любой сбой автоматически откатывает его;
- **ничто из снимка никогда не выполняется**;
- затем: хук охраны Claude переподключен к *этому* двоичному файлу машины,
  специфические для машины абсолютные пути, найденные в восстановленных конфигурациях
  агентов, предупреждаются (ваши файлы никогда не переписываются), `doctor` запускается
  встроенным, и необходимые шаги повторного входа печатаются (аутентификация никогда
  не путешествует).

Восстановления отказывают, а не угадывают — отказ восстановления оставляет проект
с точным совпадением байтов.

## `agentmod doctor`

Диагностика только для чтения, безопасная в любое время (выход 0 чистый, 3 с
результатами): состояние проекта/конфигурации/раскладки, установка и живость
хука оболочки, дрейф маршрутизации, оставшиеся переменные вне проектов,
дублирующиеся записи PATH, нарушения HOME/shim, наличие аутентификации для
каждого агента с инструкциями повторного входа, предупреждения утечки OpenCode,
состояние gstack глобально/проектное, подключение охраны Claude, риски портативности
в восстановленных конфигурациях, кандидатов секретов, записанные в существующих
снимках, материалы сеанса/журнала внутри `.agentmod-handoff/`, и совпадает ли
HEAD репозитория с наиболее новым снимком.

## Охрана Claude Bash

`agentmod init` регистрирует `agentmod guard claude-bash` как хук Claude Code
PreToolUse в локальном домашнем каталоге проекта. Это блокирует команды Bash,
которые будут писать в глобальные домашние каталоги агентов (`~/.claude`,
`~/.codex`, `~/.config/opencode`, `~/.local/share/opencode`), использовать `sudo`
или переназначать `HOME` — агент получает причину назад и может отрегулировать.
Чтения никогда не блокируются. Это одна эвристика разбора оболочки в глубину:
полезное перенаправление, а не песочница.

## Известные ограничения

Секция честности. Это свойства основных инструментов или преднамеренная область
MVP — `doctor` и созданная документация указывают их тоже.

- **macOS Keychain (Claude).** Claude Code на macOS хранит учетные данные OAuth
  в Keychain, совместно используемые всеми каталогами конфигурации. Изоляция учетной
  записи для каждого проекта невозможна на macOS — и повторный вход не требуется
  для каждого проекта. Linux/Windows используют `.credentials.json` для каждого
  домашнего каталога, что изолирует, но требует входа/копирования для каждого
  проекта.
- **OpenCode частично изолирована по умолчанию.** OpenCode не имеет единственной
  переменной домашнего каталога; его конфигурация — это цепочка слияния, которая
  все еще читает глобальный `~/.config/opencode/opencode.json`, и сеансы/хранилище/аутентификация
  находятся в глобальных каталогах XDG. `opencode.xdg_full_isolation = true`
  маршрутизирует переменные XDG для полной изоляции — но это влияет на *каждый*
  инструмент, осведомленный XDG, который вы запускаете внутри проекта. `doctor`
  сообщает обе ситуации.
- **`.claude/` уровня проекта — нативное поведение Claude.** Claude Code всегда
  читает `./.claude/` независимо от `CLAUDE_CONFIG_DIR`. Добавленная стоимость
  agentmod для Claude заключается в изоляции состояния уровня *пользователя*
  (глобальные навыки/плагины, сеансы, история); `.claude/` уровня проекта уже
  работал до agentmod.
- **Активация хука первого сеанса.** Сразу после `agentmod init` уже работающая
  оболочка не загрузила новый блок rc. Откройте новый терминал, `exec $SHELL`,
  или одиночный `eval "$(agentmod hook zsh)"` (init печатает именно это). Аналогично,
  хук bash срабатывает через `PROMPT_COMMAND` и поэтому неактивен в неинтерактивных
  скриптах bash (тот же класс ограничений, что direnv) — скрипты должны установить
  переменные явно через `eval "$(agentmod env --shell bash --activate <root>)"`,
  если им требуется маршрутизация.
- **Только глобальный bin npm находится на PATH.** `.agentmod/node/bin` — это
  единственная управляемая запись PATH. Глобальные установки pnpm/bun маршрутизируются
  в проект (`PNPM_HOME`, `BUN_INSTALL`), но их каталоги bin не добавляются в PATH.
- **Пакеты дерева восстанавливаются вручную.** `handoff restore` принимает только
  файлы `.amod`; коммитированный каталог `.agentmod-handoff/` восстанавливается,
  следуя `RESTORE.md` внутри него (эта версия не имеет средства чтения каталогов).
- **Снимки могут потребовать восстановления после восстановления.** Клон gstack
  путешествует без его `.git` (повторно запустите `agentmod install gstack --force`,
  чтобы сделать его обновляемым снова), и символические ссылки пускателя `node/bin`
  зависают, потому что `node_modules` исключен (повторно запустите `npm install -g …`
  внутри проекта).
- **Поддержка оболочки — zsh и bash.** Другие оболочки все еще могут использовать
  `agentmod env` вручную.

## FAQ

**Я продолжаю использовать `claude` / `codex` / `opencode` напрямую?**
Да. В этом суть — никаких оболочек, никаких shims, никакого `agentmod run`.

**Почему agentmod просто не меняет `HOME`?**
Переназначение `HOME` нарушает SSH, git, цепочки ключей, dotfiles и каждый
другой инструмент в оболочке. agentmod маршрутизирует только переменные,
специфичные для агента.

**Почему моя аутентификация отсутствует после восстановления?**
По замыслу — учетные данные никогда не путешествуют в снимках. Следуйте
напечатанным строкам повторного входа (или предложению копирования init)
на новой машине.

**Можно ли коммитить `.agentmod/` в git?**
Нет — init игнорирует ее (сеансы, кэши и возможно скопированная аутентификация
находятся там). Коммитьте безопасное подмножество вместо этого: `agentmod pack --for-git`.

**Чем это отличается от direnv?**
Та же модель активации (переменные окружения, охватываемые каталогом, на основе
хука подсказок, идеальное восстановление при выходе), но agentmod также знает
*что* маршрутизировать для каждого агента, создает домашние каталоги, охраняет
от глобальных записей и занимается передачей. Оба сосуществуют отлично.

**Снимок не создается с "результатами кандидата в секреты".**
Сканирование содержимого найдено материал приватного ключа в сохраненном файле.
Удалите его (или переместите в исключенное место, как `.env`), или упакуйте
все равно с `--allow-findings`, если вы принимаете его нахождение в снимке.

**Это работает на Windows?**
Код Go собирается и безопасность пути применяется для путей в стиле Windows, но
хуки оболочки нацелены на zsh/bash; Windows не протестирована в этой версии.
